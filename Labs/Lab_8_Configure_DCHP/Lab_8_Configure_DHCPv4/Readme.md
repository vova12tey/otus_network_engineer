Задачи:

Часть 1. Создание сети и настройка основных параметров устройства

Шаг 1. Создание схемы адресации
В соответствии с требованиями, исходная сеть 192.168.1.0/24 была разделена на три подсети с использованием технологии VLSM (переменная длина маски подсети):

Подсеть А (для клиентской сети VLAN 100 на R1, 58 хостов): получена подсеть 192.168.1.0/26 (маска 255.255.255.192). Первый полезный адрес (192.168.1.1) был назначен интерфейсу R1 G0/0/1.100.

Подсеть В (для управляющей сети VLAN 200, 28 хостов): получена подсеть 192.168.1.64/27 (маска 255.255.255.224). Первый полезный адрес (192.168.1.65) был назначен интерфейсу R1 G0/0/1.200, а второй полезный адрес (192.168.1.66) использован для SVI на коммутаторах. Шлюзом по умолчанию для коммутаторов был указан 192.168.1.65.

Подсеть С (для клиентской сети на R2, 12 хостов): получена подсеть 192.168.1.96/28 (маска 255.255.255.240). Первый полезный адрес (192.168.1.97) был назначен интерфейсу R2 G0/0/1.

Шаг 2. Создание сети согласно топологии

Устройства были соединены в соответствии с предоставленной топологической схемой. Маршрутизаторы были подключены друг к другу через интерфейсы G0/0/0. Соединения с коммутаторами и оконечными устройствами были установлены согласно схеме.

На маршрутизаторах R1 и R2 были настроены базовые параметры:

Устройствам были присвоены имена R1 и R2.

Отключена функция поиска DNS (no ip domain-lookup).

Установлен зашифрованный пароль class для привилегированного режима.

Установлены пароли cisco для консольного доступа и виртуальных терминалов (VTY).

Включено шифрование открытых паролей.

Создано баннерное сообщение с предупреждением о запрете несанкционированного доступа.

Установлены системные часы и сохранена текущая конфигурация в энергонезависимую память (NVRAM).

Шаг 4. Настройка маршрутизации между сетями VLAN на R1

Был включен физический интерфейс G0/0/1 маршрутизатора R1. С использованием технологии "Router-on-a-Stick" были созданы подинтерфейсы:

G0/0/1.100: с инкапсуляцией 802.1Q для VLAN 100 и первым полезным адресом 192.168.1.1/26.

G0/0/1.200: с инкапсуляцией 802.1Q для VLAN 200 и первым полезным адресом 192.168.1.65/27.

G0/0/1.1000: с инкапсуляцией 802.1Q для native VLAN 1000, без назначения IP-адреса. Каждому интерфейсу было добавлено текстовое описание.



Шаг 5. Настройка интерфейсов и статической маршрутизации
На маршрутизаторе R2 был настроен интерфейс G0/0/1 с адресом 192.168.1.97/28. На обоих маршрутизаторах были настроены транковые интерфейсы G0/0/0 (R1 – 10.0.0.1/30, R2 – 10.0.0.2/30).
Для обеспечения сквозной связности между сетями были настроены статические маршруты по умолчанию: на R1 был указан маршрут через 10.0.0.2, а на R2 — через 10.0.0.1. Связность была успешно проверена с помощью эхо-запросов от R1 к интерфейсу R2 G0/0/1.

Шаг 6. Настройка базовых параметров коммутаторов
Аналогично маршрутизаторам, на коммутаторах S1 и S2 были настроены базовые параметры: присвоены имена, отключен DNS, установлены зашифрованные пароли и баннеры, настроены консольные и виртуальные каналы.

Шаг 7. Создание сетей VLAN на коммутаторе S1
На коммутаторе S1 были созданы следующие VLAN: VLAN 100 (Clients), VLAN 200 (Management), VLAN 999 (Parking_Lot) и VLAN 1000 (Native). Для управления был настроен интерфейс SVI для VLAN 200 с IP-адресом 192.168.1.66/27 и шлюзом по умолчанию 192.168.1.65. На S2 управляющий интерфейс SVI был настроен в VLAN 1 с тем же IP-адресом 192.168.1.66/27 и шлюзом по умолчанию, так как обе сети управления находятся в одной подсети. Все неиспользуемые порты на S1 были переведены в VLAN 999 и административно отключены. На S2 неиспользуемые порты были административно отключены.

Шаг 8. Назначение сетей VLAN соответствующим интерфейсам коммутатора
Порт F0/6 на S1 был настроен как порт доступа и назначен в VLAN 100 для подключения PC-A. Порт F0/18 на S2 был настроен как порт доступа в VLAN 1 для подключения PC-B.

Ответ на вопрос: Интерфейс F0/5 на S1 указан в VLAN 1 только в выводе команды show vlan brief, так как он был настроен в режиме trunk. Транковые порты по умолчанию отображаются в VLAN 1, однако фактически они пропускают трафик всех настроенных на них VLAN.

Шаг 9. Вручную настроен транк 802.1Q на S1
Интерфейс F0/5 на S1 был переведен в режим статической магистрали (switchport mode trunk). В качестве собственной (native) VLAN была установлена VLAN 1000. Также была ограничена пропускная способность транка до VLAN 100, 200 и 1000.

Часть 2. Настройка и проверка двух серверов DHCPv4 на R1
Шаг 1. Настройка пулов DHCPv4
На маршрутизаторе R1 была настроена служба DHCP для обслуживания двух подсетей (Подсеть А и Подсеть С):

Сначала из пулов были исключены первые 5 IP-адресов в каждой подсети (в Подсети А — с 192.168.1.1 по 192.168.1.5, в Подсети С — с 192.168.1.97 по 192.168.1.101).

Был создан пул POOL_A для подсети 192.168.1.0/26. В нем были указаны шлюз по умолчанию (192.168.1.1), доменное имя (CCNA-lab.com) и время аренды на 2 дня 12 часов 30 минут.

Затем был создан пул POOL_C для подсети 192.168.1.96/28. В качестве шлюза по умолчанию был указан адрес 192.168.1.97 (интерфейс R2), задано то же доменное имя и время аренды.

Шаг 2. Сохранение конфигурации
Все изменения конфигурации маршрутизатора R1 были сохранены в файл загрузочной конфигурации.

Шаг 3. Проверка конфигурации сервера DHCPv4
Были выполнены проверочные команды на R1: show ip dhcp pool (для просмотра пулов и статистики), show ip dhcp bindings (для проверки текущих назначений) и show ip dhcp server statistics (для просмотра статистики DHCP-сервера).

Шаг 4. Попытка получения IP-адреса от DHCP на PC-A
На компьютере PC-A в командной строке была выполнена команда ipconfig /renew. После успешного обновления был проверен полученный адрес командой ipconfig /all. Компьютер PC-A получил IP-адрес из пула POOL_A (например, 192.168.1.6, так как первые 5 были исключены), а также корректный шлюз по умолчанию. Работа DHCP была проверена эхо-запросами до шлюза 192.168.1.1.

Часть 3. Настройка и проверка DHCP-ретрансляции на R2

Шаг 1. Настройка R2 в качестве агента DHCP-ретрансляции

Для того чтобы запросы от PC-B (находящегося в сети 192.168.1.96/28) могли достичь сервера DHCPv4, размещенного на R1, на маршрутизаторе R2 была настроена ретрансляция. На интерфейсе G0/0/1 (обращенном к PC-B) была введена команда ip helper-address 10.0.0.1, указывающая на IP-адрес интерфейса R1 G0/0/0. Конфигурация была сохранена.

Шаг 2. Попытка получения IP-адреса от DHCP на PC-B

На компьютере PC-B была выполнена команда ipconfig /release и ipconfig /renew. После завершения процесса обновления командой ipconfig /all было проверено присвоение IP-адреса. PC-B успешно получил IP-адрес из сети 192.168.1.96/28 (например, 192.168.1.102), шлюз по умолчанию 192.168.1.97 и доменное имя CCNA-lab.com.
Связность между сетями была проверена эхо-запросами от PC-B до интерфейса R1 G0/0/1 (ping 192.168.1.65) — пинги прошли успешно.

На маршрутизаторе R1 с помощью команд show ip dhcp binding были просмотрены установленные назначения адресов из пула POOL_C, а с помощью show ip dhcp server statistics подтвержден обмен DHCP-сообщениями через ретранслятор.


В ходе выполнения лабораторной работы была развернута сеть, состоящая из двух маршрутизаторов, двух коммутаторов и двух оконечных устройств. Основной целью являлось внедрение службы DHCPv4 на маршрутизаторе R1 для автоматического назначения IP-адресов хостам в разных подсетях, а также настройка ретрансляции DHCP-запросов через маршрутизатор R2.
