Часть 1. Создание сети и настройка основных параметров устройства

Шаг 1. Создание схемы адресации
В соответствии с требованиями, исходная сеть 192.168.1.0/24 была разделена на три подсети с использованием технологии VLSM (переменная длина маски подсети):

Подсеть А (для клиентской сети VLAN 100 на R1, 58 хостов): получена подсеть 192.168.1.0/26 (маска 255.255.255.192). Первый полезный адрес (192.168.1.1) был назначен интерфейсу R1 G0/0/1.100.

Подсеть В (для управляющей сети VLAN 200, 28 хостов): получена подсеть 192.168.1.64/27 (маска 255.255.255.224). Первый полезный адрес (192.168.1.65) был назначен интерфейсу R1 G0/0/1.200, а второй полезный адрес (192.168.1.66) использован для SVI на коммутаторах. Шлюзом по умолчанию для коммутаторов был указан 192.168.1.65.

Подсеть С (для клиентской сети на R2, 12 хостов): получена подсеть 192.168.1.96/28 (маска 255.255.255.240). Первый полезный адрес (192.168.1.97) был назначен интерфейсу R2 G0/0/1.

Шаг 2. Создание сети согласно топологии

Устройства были соединены в соответствии с предоставленной топологической схемой. Маршрутизаторы были подключены друг к другу через интерфейсы G0/0/0. Соединения с коммутаторами и оконечными устройствами были установлены согласно схеме.

Шаг 4. Настройка маршрутизации между сетями VLAN на R1

Был включен физический интерфейс G0/0/1 маршрутизатора R1. С использованием технологии "Router-on-a-Stick" были созданы подинтерфейсы:

G0/0/1.100: с инкапсуляцией 802.1Q для VLAN 100 и первым полезным адресом 192.168.1.1/26.

G0/0/1.200: с инкапсуляцией 802.1Q для VLAN 200 и первым полезным адресом 192.168.1.65/27.

G0/0/1.1000: с инкапсуляцией 802.1Q для native VLAN 1000, без назначения IP-адреса. Каждому интерфейсу было добавлено текстовое описание.



Шаг 5. Настройка интерфейсов и статической маршрутизации
На маршрутизаторе R2 был настроен интерфейс G0/0/1 с адресом 192.168.1.97/28. На обоих маршрутизаторах были настроены транковые интерфейсы G0/0/0 (R1 – 10.0.0.1/30, R2 – 10.0.0.2/30).
Для обеспечения сквозной связности между сетями были настроены статические маршруты по умолчанию: на R1 был указан маршрут через 10.0.0.2, а на R2 — через 10.0.0.1. Связность была успешно проверена с помощью эхо-запросов от R1 к интерфейсу R2 G0/0/1.

Шаг 6. Настройка базовых параметров коммутаторов
Аналогично маршрутизаторам, на коммутаторах S1 и S2 были настроены базовые параметры: присвоены имена, отключен DNS, установлены зашифрованные пароли и баннеры, настроены консольные и виртуальные каналы.



Шаг 7. Создание сетей VLAN на коммутаторе S1
На коммутаторе S1 были созданы следующие VLAN: VLAN 100 (Clients), VLAN 200 (Management), VLAN 999 (Parking_Lot) и VLAN 1000 (Native). Для управления был настроен интерфейс SVI для VLAN 200 с IP-адресом 192.168.1.66/27 и шлюзом по умолчанию 192.168.1.65. На S2 управляющий интерфейс SVI был настроен в VLAN 1 с тем же IP-адресом 192.168.1.66/27 и шлюзом по умолчанию, так как обе сети управления находятся в одной подсети. Все неиспользуемые порты на S1 были переведены в VLAN 999 и административно отключены. На S2 неиспользуемые порты были административно отключены.

Вопрос: "Почему интерфейс F0/5 указан в VLAN 1?"
Ответ: Физически F0/5 настроен как транковый порт (не access). В выводе show vlan brief транковые порты Cisco по умолчанию отображаются в VLAN 1, даже если фактически через них проходят все VLAN. Это нормальное поведение STP и DTP.


Часть 2. Настройка и проверка двух серверов DHCPv4 на R1

Шаг 1. Настройка пулов DHCPv4 на R1

На маршрутизаторе R1 была настроена служба DHCP для обслуживания двух подсетей (Подсеть А и Подсеть С).

Сначала из пулов были исключены первые 5 IP-адресов в каждой подсети (с 192.168.1.1 по 192.168.1.5 и с 192.168.1.97 по 192.168.1.101).

Был создан пул POOL_A для подсети 192.168.1.0/26. В нём были указаны шлюз по умолчанию (192.168.1.1) и доменное имя (CCNA-lab.com). Установка точного времени аренды не была выполнена, так как используемая версия эмулятора (Packet Tracer) не поддерживала синтаксис команды lease для данного образа маршрутизатора. DHCP-сервер использовал настройки времени аренды по умолчанию.

Затем был создан пул POOL_C для подсети 192.168.1.96/28. В качестве шлюза по умолчанию был указан адрес 192.168.1.97, задано то же доменное имя.

Шаг 3 и 4. Проверка конфигурации сервера DHCPv4 и получение адреса на PC-A

На компьютере PC-A были выполнены команды ipconfig /release и ipconfig /renew. После успешного обновления конфигурации ПК получил IPv4-адрес 192.168.1.6 с маской 255.255.255.192, шлюз по умолчанию 192.168.1.1 и DNS-суффикс CCNA-lab.com.

С помощью команды ping 192.168.1.1 была успешно проверена связь с интерфейсом маршрутизатора. Ответы от R1 были получены без потерь пакетов.

На маршрутизаторе R1 с помощью команды show ip dhcp pool была подтверждена выдача одного адреса из пула POOL_A (Leased addresses: 1). 
Команда show ip dhcp bindings не была использована из-за ограничений используемого эмулятора (Packet Tracer не поддерживает эту команду на данной модели маршрутизатора). Также в выводе show ip dhcp pool наблюдалось отображение исключённых адресов как 2 (вместо 5), что является артефактом отображения Packet Tracer, однако адрес 192.168.1.6 был выдан корректно, что подтверждает правильную работу настроенного исключения диапазона.

Вопрос: "Какой IP-адрес был бы у ПК, если бы он был подключен к сети с помощью DHCP?"

Ответ: Любой свободный IP из пула, начиная с 192.168.1.6 (так как адреса с 1 по 5 исключены). Например, 192.168.1.6.

Часть 3. Настройка и проверка DHCP-ретрансляции на R2

Шаг 1. Настройка R2 в качестве агента DHCP-ретрансляции
На интерфейсе R2 G0/0/1 была настроена команда ip helper-address 10.0.0.1, указывающая на адрес DHCP-сервера (R1). Конфигурация была сохранена.

Диагностика и устранение неполадок
После настройки PC-B выдавал APIPA-адрес (169.254.238.29), а интерфейс R2 G0/0/1 находился в состоянии up/down. Проверка на коммутаторе S2 показала, что порт F0/5 (ведущий к R2) находился в режиме dynamic auto, из-за чего канальный протокол не поднимался с маршрутизатором. Порт F0/5 на S2 был принудительно переведён в режим доступа (switchport mode access), назначен в VLAN 1 и активирован (no shutdown). После этого на R2 интерфейс G0/0/1 поднялся в статус up/up (подтверждено командой show ip interface brief и сообщением %LINEPROTO-5-UPDOWN).

Шаг 2. Проверка получения IP-адреса на PC-B
После восстановления канального соединения на PC-B была выполнена команда ipconfig /renew. Устройство успешно получило IPv4-адрес 192.168.1.102 с маской 255.255.255.240, шлюз по умолчанию 192.168.1.97 и DNS-суффикс CCNA-lab.com. Эхо-запрос до интерфейса R1 192.168.1.65 прошёл успешно. На маршрутизаторе R1 команда show ip dhcp pool подтвердила выдачу адреса из пула POOL_C (Leased addresses: 1).
